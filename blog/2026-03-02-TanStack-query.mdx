The analysis identifies 84 critical-band functions, with the top hotspots concentrated in query persistence, code transformation, and developer tooling—indicating that active development in foundational infrastructure is outpacing structural simplification.

The table below ranks functions by **activity-weighted risk** — a score that multiplies structural complexity by recent commit frequency. A function that is both hard to understand (high cyclomatic complexity) and actively changing is a higher priority than one that is complex but untouched. CC = cyclomatic complexity (independent execution paths); ND = max nesting depth; FO = fan-out (distinct callees).

## Hotspot Analysis

### `experimental_createQueryPersister` — packages/query-persist-client-core/src/createPersister.ts

This factory function initializes query persistence logic and ranks as the highest-priority hotspot with an activity_risk of 16.82. It combines moderate cyclomatic complexity (CC 14), deep nesting (ND 6), and broad fan-out (FO 27), meaning it orchestrates 27 distinct functions—a significant coupling footprint. The patterns of complex_branching, deeply_nested logic, and exit_heavy paths (multiple return routes) suggest the initialization sequence has grown to handle too many concerns, and active commits indicate it continues to evolve without structural simplification.

**Recommendation:** Map the 27 callees to identify clusters of related logic (e.g., event handling, storage, validation), then extract sub-functions to reduce fan-out. Before refactoring, add characterization tests covering each of the ND 6 nesting levels to prevent regression.

### `restoreQueries` — packages/query-persist-client-core/src/createPersister.ts

This sibling function in the same file tackles query restoration with activity_risk 15.56 and carries CC 19 (moderate-to-high branching). Its ND 6 matches the parent function, signaling deeply nested conditional chains; exit_heavy and complex_branching patterns indicate multiple decision paths that each require test coverage. The lower fan-out (FO 8) is positive, but the combination of high CC and deep nesting in an actively maintained code path raises regression risk.

**Recommendation:** Extract the CC 19 branching logic into smaller decision-tree functions (one per major conditional block), then add unit tests for each path before merging future changes to this restoration sequence.

### `Explorer` — packages/query-devtools/src/Explorer.tsx

The devtools Explorer component is the most broadly coupled function in the dataset, with activity_risk 15.47 driven by CC 26 and an exceptionally high fan-out of 38 distinct function calls. This is a god_function pattern—it likely orchestrates rendering, event handling, and state management across the entire devtools UI. The high CC combined with high fan-out means changes here have a wide blast radius and affect many downstream consumers.

**Recommendation:** Decompose Explorer into smaller, single-responsibility components (e.g., a QueryTree component, a QueryDetails component, a Controls component), each with its own fan-out cluster. This reduces the blast radius and makes the devtools easier to test and maintain.
