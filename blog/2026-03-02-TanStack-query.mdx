The table below ranks functions by **activity-weighted risk** — a score that multiplies structural complexity by recent commit frequency. A function that is both hard to understand (high cyclomatic complexity) and actively changing is a higher priority than one that is complex but untouched. CC = cyclomatic complexity (independent execution paths); ND = max nesting depth; FO = fan-out (distinct callees).

## Hotspot Analysis

### `experimental_createQueryPersister` — packages/query-persist-client-core/src/createPersister.ts

This function orchestrates query persistence initialization and likely manages serialization, cache hydration, and subscription logic. With CC 14, ND 6, and FO 27, it combines moderate-to-high cyclomatic branching with deep nesting that makes control flow difficult to trace, plus broad fan-out that creates coupling to 27 distinct downstream functions. The activity_risk score of 16.82 (driven by LRS 15.79) signals active modification to an already-complex core function—each change carries regression risk across the persistence subsystem.

**Recommendation:** Before next refactor, map the 27 callees to identify whether fan-out can be consolidated via dependency injection or facade patterns. Extract nested conditional blocks (ND 6) into named helper functions to reduce cognitive load and test surface; prioritize characterization tests covering each of the 14 independent code paths.

### `restoreQueries` — packages/query-persist-client-core/src/createPersister.ts

A sibling function in the same persistence module, restoreQueries manages hydration of cached query state with CC 19 and ND 6. The deep nesting and multiple exit paths (exit_heavy pattern) mean test coverage must account for many early returns and conditional branches. Activity_risk 15.56 reflects steady modification to restoration logic—a critical path where bugs directly impact cache correctness and user experience.

**Recommendation:** Audit all return paths and early-exit conditions to ensure each is covered by integration tests. Consider extracting validation and error-handling blocks into separate helper functions to flatten ND from 6 to 3–4, improving readability without changing behavior.

### `Explorer` — packages/query-devtools/src/Explorer.tsx

The devtools Explorer component is a high-fan-out hub (FO 38) with CC 26 branching complexity, orchestrating state display, filtering, and interaction across the DevTools UI. Its activity_risk of 15.44 and exit_heavy, god_function patterns reveal that despite moderate nesting (ND 3), the sheer number of callees and independent logic paths creates a wide blast radius—changes here propagate across DevTools interactions and monitoring features.

**Recommendation:** Decompose Explorer into smaller, single-responsibility components (e.g., ExplorerHeader, ExplorerQuery, ExplorerActions) to distribute the 38 fan-out calls and reduce CC per component. Review which state-management concerns belong in a custom hook versus component logic.

## Key Takeaways

- The persistence core (`createPersister.ts`) contains two critical hotspots with activity_risk >15.5, both in the `complex-active` quadrant. Prioritize extracting nested conditionals (ND 6) and characterization tests before making further modifications to either `experimental_createQueryPersister` or `restoreQueries`.
- Explorer (devtools) exhibits god_function antipattern with FO 38—the highest fan-out among the top 5. Each change risks unexpected side effects across monitoring and interaction logic; decompose into feature-focused child components to reduce coupling and testing burden.
- All five hotspots display exit_heavy patterns (multiple return paths), indicating that test suites must cover many conditional branches. Audit existing coverage for CC 14–26 functions; add integration tests targeting rare branches and error conditions before undertaking refactoring.
