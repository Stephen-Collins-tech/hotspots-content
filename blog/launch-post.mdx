Most teams refactor by gut feel. Some files look messy, while others quietly cause incidents because they're both complex and frequently changed. Hotspots finds that overlap — your hotspots — so you focus effort where it matters.

What makes a hotspot?

- Local Risk Score (LRS): a composite of cyclomatic complexity (CC), nesting depth (ND), fan‑out (FO), and non‑structured exits (NS).
- Activity risk: git churn, touch frequency, recency, and call‑graph influence (fan‑in/out).

Why this works:

- Targeted: Prioritize the small set of functions creating outsized risk.
- Preventative: Block complexity regressions in CI with policy checks.
- Explainable: Driver labels and triage quadrants make action obvious.

Try it fast:

```bash
hotspots analyze src/
hotspots analyze src/ --mode delta --policy
hotspots analyze src/ --mode snapshot --format html --output .hotspots/report.html
```

What you'll see:

- A ranked list of functions with risk bands and driver labels
- Triage groups (fire/debt/watch/ok) to decide what to do first
- Optional HTML report for sharing and CI deltas for gating PRs

Docs and examples at https://docs.hotspots.dev and source at https://github.com/Stephen-Collins-tech/hotspots.
