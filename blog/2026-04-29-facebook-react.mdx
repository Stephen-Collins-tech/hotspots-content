Of React's 3,529 analyzed functions, 402 are rated critical — and the five highest-risk hotspots are all concentrated in two actively developed subsystems: the React Compiler's HIR builder (`BuildHIR.ts`) and the `eslint-plugin-react-hooks` exhaustive-deps rule (`ExhaustiveDeps.ts`). The top-ranked function, `lowerStatement`, sits in the "fire" quadrant with a cyclomatic complexity of 150 and a recent activity score of 20.2, meaning it is both structurally extreme and actively changing right now — a live regression risk, not a backlog cleanup item. Every one of the top five functions shares the same pattern fingerprint: god-function, complex branching, deep nesting, exit-heavy, and long — a consistent structural signature across two of React's most actively evolved components.

The table below ranks functions by **activity-weighted risk** — a score that multiplies structural complexity by recent commit frequency. A function that is both hard to understand (high cyclomatic complexity) and actively changing is a higher priority than one that is complex but untouched. CC = cyclomatic complexity (independent execution paths); ND = max nesting depth; FO = fan-out (distinct callees).

## Large Repo Analysis

react is a large repository. To stay within memory constraints, this analysis used **hybrid touch mode**: structural complexity — CC, ND, FO — is measured precisely for every function. Git activity is tracked at the function level (via `git log -L`) only for files with 5 or more commits in the last 30 days; other files use a file-level approximation. Rankings therefore surface functions that are both structurally complex and in the most actively-changing parts of the codebase. Dormant code with high structural complexity will rank lower than it would under a full per-function analysis — to surface it, run `hotspots analyze . --per-function-touches` on a machine with sufficient memory.

## Hotspot Analysis

### `lowerStatement` — compiler/packages/babel-plugin-react-compiler/src/HIR/BuildHIR.ts

Based on its name and location in the HIR (High-level Intermediate Representation) builder, `lowerStatement` almost certainly handles the translation of JavaScript/TypeScript statements into the compiler's internal IR — a central pass that must branch on every statement type the language supports. A cyclomatic complexity of 150 means 150 independent execution paths, each a required test case and a potential bug surface; a max nesting depth of 8 pushes well past the point where local reasoning is reliable. Its fan-out of 63 distinct callees means a change to this function can propagate through a wide swath of the compiler pipeline. Sitting in the "fire" quadrant with a recent activity score of 20.2 and an activity risk of 25.4 — the highest in the repo — this is a live regression risk: complex logic is being actively modified right now.

**Recommendation:** Before any further feature work touches this function, add a comprehensive characterization test suite covering the major statement-type branches; then apply extract-method refactoring to isolate each statement category (e.g., loops, conditionals, try/catch, declarations) into dedicated lower* helpers, reducing the monolithic CC-150 surface into individually testable units.

### `lowerExpression` — compiler/packages/babel-plugin-react-compiler/src/HIR/BuildHIR.ts

`lowerExpression` likely handles the parallel translation task for expressions within the same HIR compilation pass. With a cyclomatic complexity of 141 and a fan-out of 76 — the highest fan-out among all five hotspots — it is broadly coupled to more downstream components than even `lowerStatement`. Its nesting depth of 6, combined with the exit-heavy pattern, means the function has many early-return paths that are difficult to cover uniformly with tests. Its recent activity score of 18.6 and "fire" quadrant placement confirm it is being actively modified alongside `lowerStatement`, compounding the regression risk: two CC-100+ functions in the same file are changing in parallel.

**Recommendation:** Audit the 76 callees via fan-out analysis to map the blast radius, then prioritize extracting expression-category sub-functions (e.g., member expressions, call expressions, literals) as independent units — this directly reduces both the CC and the coupling surface before the next compiler feature lands.

### `lowerAssignment` — compiler/packages/babel-plugin-react-compiler/src/HIR/BuildHIR.ts

`lowerAssignment` handles the translation of assignment expressions into the compiler's HIR — a narrower task than `lowerStatement` but still responsible for branching across every assignment form JavaScript supports (simple assignment, destructuring, compound assignment, optional chaining targets). A cyclomatic complexity of 72 and a nesting depth of 6 indicate significant branching, and its fan-out of 42 means changes propagate to 42 downstream components in the compiler pipeline. It sits alongside `lowerStatement` and `lowerExpression` in the "fire" quadrant, confirming it is actively modified as the compiler evolves.

**Recommendation:** Since `lowerAssignment` shares a file with the two higher-ranked hotspots, coordinate any refactoring effort across all three — changes to assignment lowering will likely surface shared utilities that can also simplify the broader `lowerStatement` and `lowerExpression` surfaces. Start by extracting destructuring and compound-assignment branches as independent helpers with clear input contracts.

### `visitFunctionWithDependencies` — packages/eslint-plugin-react-hooks/src/rules/ExhaustiveDeps.ts

From its name and location in the exhaustive-deps lint rule, `visitFunctionWithDependencies` likely orchestrates the core analysis that walks a React hook's callback, resolves its dependency scope, and validates the dependency array — the most logic-dense part of the rule. A cyclomatic complexity of 131 combined with a max nesting depth of 10 — the deepest in the dataset — makes this function particularly hard to reason about locally; ND 10 means the innermost logic is ten control-structure levels removed from the function signature. With a fan-out of 74 and a recent activity score of 20.2 in the "fire" quadrant, structural complexity and active development pressure are coinciding right now, making untested edge-case regressions a concrete near-term risk.

**Recommendation:** The nesting depth of 10 is the most urgent signal here: identify the innermost nested blocks and extract them into named helper functions with clear contracts, which will also make the dependency-resolution logic independently unit-testable without exercising the full visitor path.

### `create` — packages/eslint-plugin-react-hooks/src/rules/ExhaustiveDeps.ts

`create` is the rule factory for the exhaustive-deps lint rule — the entry point that Babel/ESLint calls to instantiate the rule's visitor object. With a cyclomatic complexity of 125, a nesting depth of 10, and a fan-out of 89 — the highest callout count of any function in this dataset — it is both a structural hotspot and a coupling hub for the entire rule. Nearly every dependency-tracking path eventually connects through this function. Despite its role as an initializer, its complexity suggests it has accumulated significant inline logic that would be better expressed as named sub-visitors or helper functions. Its "fire" quadrant placement confirms active recent modification.

**Recommendation:** The fan-out of 89 is the primary concern: audit which of the 89 callees are cohesive sub-responsibilities (e.g., scope analysis, dependency tracking, write detection) and extract each into a dedicated visitor or helper. This reduces both the CC and the coupling surface, and makes individual rule behaviors independently testable without instantiating the full rule factory.
